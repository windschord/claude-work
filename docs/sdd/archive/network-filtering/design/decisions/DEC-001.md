# DEC-001: iptables DOCKER-USER chain方式の採用

## ステータス

承認済

## コンテキスト

Dockerコンテナからの外部ネットワーク通信をフィルタリングする実装方式を選定する必要がある。

## 検討した選択肢

### A: iptables DOCKER-USER chain方式（採用）

Docker公式が提供する `DOCKER-USER` chain を利用し、ユーザー定義のフィルタリングルールを適用する。

**メリット**:
- Docker公式がサポートする方式であり、Dockerのネットワーキングと干渉しない
- DOCKER-USER chainはDockerによって自動的にFORWARD chainの先頭で評価される
- Dockerのバージョンアップに対して安定的
- Docker単体/Docker Compose両方で同じ方式が使える

**デメリット**:
- iptablesコマンドのroot権限（またはCAP_NET_ADMIN）が必要
- ドメイン名ベースのフィルタリングにはDNS解決が必要（IPアドレスレベルの制御）

### B: Docker --internal ネットワーク + プロキシ方式

Docker networkの`--internal`フラグで外部通信を完全遮断し、プロキシコンテナ経由で選択的に通信を許可する。

**メリット**:
- Docker nativeの機能でネットワーク分離が可能
- HTTP/HTTPSレベルのドメインフィルタリングが容易

**デメリット**:
- プロキシコンテナの管理が必要（リソース消費増加）
- HTTP/HTTPS以外のプロトコルの制御が困難
- 設定の複雑性が高い

### C: Docker Compose network設定のみ

Docker Composeのnetwork設定（internal, external等）で制御する。

**メリット**:
- 設定ファイルベースでシンプル

**デメリット**:
- 細かいルール（ドメイン/IP + ポート）の制御が不可能
- Docker Compose環境以外では使えない

## 決定

**A: iptables DOCKER-USER chain方式** を採用する。

## 根拠

1. Docker公式の推奨方式であり、安定性と互換性が保証される
2. IP + ポートレベルの細かいフィルタリングが可能
3. Docker単体/Docker Compose両環境で統一的に使える
4. プロキシコンテナ等の追加リソースが不要
5. 既存のDockerAdapterの`CapDrop: ['ALL']`設定と組み合わせることで、コンテナ内からのバイパスを防止できる

## 影響

- ClaudeWorkサーバーがiptablesコマンドを実行できる権限が必要
  - HOST環境: root権限またはsudo設定
  - Docker Compose環境: `cap_add: [NET_ADMIN]` + `network_mode: host`（iptables操作用）
    - または `nsenter` 経由でホストのnetwork namespaceにアクセス
- コンテナ起動フローにiptablesルール適用ステップが追加される
