# DEC-001: 拡張Dockerfileの継承パターン

## ステータス

**ステータス**: 承認済
**決定日**: 2026-03-03

## コンテキスト

ClaudeWorkの実行環境Dockerイメージ（`docker/Dockerfile`）はNode.js + Claude Codeのみで構成されている。Go, Python, Rust, C++等のプロジェクトで作業するには各言語の開発ツールが必要だが、全てを1イメージに入れるとサイズが肥大化する。各言語拡張をどのようにDockerfileとして構成するかを決定する必要がある。

### 現状の課題
- 基本イメージに全言語を含めるとサイズが肥大化する
- 各言語Dockerfileに基本イメージの内容を重複記載すると管理が困難

### 制約条件
- 基本のDockerfileは変更しない
- 将来的に複合イメージ（例: Python + Chrome DevTools）が必要

## 検討した選択肢

### 選択肢A: ARG BASE_IMAGE継承パターン

**概要**: 全拡張DockerfileでARG/FROM構文を使い、任意のイメージを継承可能にする

```dockerfile
ARG BASE_IMAGE=ghcr.io/windschord/claude-work-sandbox:latest
FROM ${BASE_IMAGE}
```

**メリット**:
- 基本イメージの変更が自動的に全拡張に反映
- 複合イメージ対応が自然（BASE_IMAGEを別拡張に差し替えるだけ）
- 各拡張は差分のみ記述（重複ゼロ）

**デメリット**:
- レイヤー数が増加（ベース + 拡張の合計）
- 拡張の積み重ねによるイメージサイズの累積

### 選択肢B: マルチステージビルド

**概要**: 1つのDockerfileにマルチステージで全バリアントを定義

**メリット**:
- 単一Dockerfileで管理
- ステージ間でのファイルコピーが可能

**デメリット**:
- Dockerfileが巨大化
- CIでの選択的ビルドが煩雑
- 複合イメージの構成が困難

### 選択肢C: 独立Dockerfile（重複あり）

**概要**: 各言語ごとに完全に独立したDockerfileを作成

**メリット**:
- 各Dockerfileが自己完結
- 依存関係なし

**デメリット**:
- 基本イメージの変更を全ファイルに反映する必要がある
- 大量の重複コード
- 管理コストが高い

## 決定

**選択**: 選択肢A: ARG BASE_IMAGE継承パターン

**根拠**:

1. 基本イメージの変更が自動的に全拡張に反映され、管理コストが最小
2. Chrome DevTools拡張（PR#184）で既にこのパターンを採用・動作確認済み
3. 将来の複合イメージ対応が自然な拡張で実現可能
4. レイヤー数の増加はDocker layer cachingにより実用上問題にならない

## 影響

### 影響を受けるコンポーネント
- 全拡張Dockerfile: このパターンに準拠して作成
- CI/CD: BASE_IMAGEをbuild-argsで渡す仕組みが必要

### 必要な変更
- 各拡張Dockerfileの先頭に `ARG BASE_IMAGE` / `FROM ${BASE_IMAGE}` を記述
- CI/CDワークフローでビルド順序の依存関係を設定（base -> extensions）
