# DEC-002: ディレクトリ構成とファイル配置

## ステータス

**ステータス**: 承認済
**決定日**: 2026-03-03

## コンテキスト

複数の拡張Dockerfileを追加するにあたり、ファイルの配置場所を決定する必要がある。基本Dockerfileは `docker/Dockerfile` に存在し、Chrome DevTools拡張は現在 `docker/Dockerfile.chrome-devtools` にある。

### 現状の課題
- 拡張ファイルが増えると `docker/` ディレクトリが煩雑になる
- 基本Dockerfileと拡張Dockerfileの区別がつきにくい

### 制約条件
- 基本Dockerfileの位置 (`docker/Dockerfile`) は変更しない
- CI/CDのビルドコンテキスト（`context: docker`）はそのまま使用

## 検討した選択肢

### 選択肢A: `docker/extensions/` サブディレクトリ

**概要**: 全拡張を `docker/extensions/` に配置

```
docker/
├── Dockerfile
└── extensions/
    ├── Dockerfile.python
    ├── Dockerfile.golang
    ├── Dockerfile.rust
    ├── Dockerfile.cpp
    └── Dockerfile.chrome-devtools
```

**メリット**:
- 基本と拡張の明確な分離
- ディレクトリ構造が直感的
- 拡張の追加・削除が容易

**デメリット**:
- 既存の `docker/Dockerfile.chrome-devtools` の移動が必要

### 選択肢B: `docker/` フラットディレクトリ

**概要**: 全Dockerfileを `docker/` 直下に配置

```
docker/
├── Dockerfile
├── Dockerfile.python
├── Dockerfile.golang
├── Dockerfile.rust
├── Dockerfile.cpp
└── Dockerfile.chrome-devtools
```

**メリット**:
- ディレクトリ階層が浅い
- 既存のChrome DevToolsファイル移動不要

**デメリット**:
- ファイル数増加でディレクトリが煩雑
- 基本と拡張の区別が命名規則に依存

### 選択肢C: 言語ごとのサブディレクトリ

**概要**: 各言語にサブディレクトリを作成

```
docker/
├── Dockerfile
├── python/
│   └── Dockerfile
├── golang/
│   └── Dockerfile
└── ...
```

**メリット**:
- 言語固有の設定ファイルも同居可能

**デメリット**:
- ディレクトリが深い
- Dockerfileが全て同名で紛らわしい
- ビルドコンテキストの指定が複雑化

## 決定

**選択**: 選択肢A: `docker/extensions/` サブディレクトリ

**根拠**:

1. 基本イメージと拡張イメージの役割が明確に分離される
2. 拡張の追加時に `docker/extensions/` に新ファイルを置くだけで完了
3. CI/CDのmatrix定義で `docker/extensions/Dockerfile.*` のパターンが使いやすい
4. ユーザーからの指示で「各言語の追加拡張の扱い」が求められており、サブディレクトリでの分離が適切

## 影響

### 影響を受けるコンポーネント
- `docker/Dockerfile.chrome-devtools`: `docker/extensions/` に移動
- `.github/workflows/docker-publish.yml`: ファイルパスの更新

### 必要な変更
- `docker/Dockerfile.chrome-devtools` を `docker/extensions/Dockerfile.chrome-devtools` に移動
- PR#184のレビュー/マージ後に移動を実施
- CI/CDのmatrix設定でパスを更新
