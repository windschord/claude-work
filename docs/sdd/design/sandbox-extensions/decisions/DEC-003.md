# DEC-003: CI/CDの2段階マトリクスビルド戦略

## ステータス

**ステータス**: 承認済
**決定日**: 2026-03-03

## コンテキスト

基本イメージと複数の拡張イメージをCI/CDでビルド・プッシュする方法を決定する必要がある。拡張イメージは基本イメージに依存するため、ビルド順序の制御が必要。

### 現状の課題
- 既存のワークフローは基本イメージのみのシンプルな構成
- 拡張イメージの追加に伴い、ビルド構成の拡張が必要

### 制約条件
- GitHub Actionsのジョブ間依存（`needs:`）を使用
- マルチプラットフォーム（linux/amd64, linux/arm64）対応が必要
- PRではビルドのみ、pushはmainブランチのみ

## 検討した選択肢

### 選択肢A: 2段階マトリクスビルド

**概要**: `build-base` ジョブ完了後に `build-extensions` ジョブをmatrix strategyで並列実行

**メリット**:
- 拡張ビルドが並列化されビルド時間短縮
- `needs:` で依存関係が明確
- 新しい拡張はmatrixに1行追加するだけ
- `fail-fast: false` で1つの失敗が他に影響しない

**デメリット**:
- ジョブ間でのイメージ参照にsha-tagの受け渡しが必要

### 選択肢B: 逐次ビルド

**概要**: 全イメージを1つのジョブで順次ビルド

**メリット**:
- シンプルなワークフロー構成
- ジョブ間のデータ受け渡し不要

**デメリット**:
- ビルド時間が長い（全拡張が逐次）
- 1つの拡張の失敗で後続が全てスキップ
- ワークフローが長大化

### 選択肢C: 別ワークフローに分離

**概要**: 基本イメージと拡張イメージで別ワークフローに分ける

**メリット**:
- ワークフローが小さく管理しやすい
- 基本イメージのみの変更時に拡張ビルド不要

**デメリット**:
- ワークフロー間の連携が複雑（`workflow_run` トリガー等）
- トリガー条件の管理が煩雑
- デバッグが困難

## 決定

**選択**: 選択肢A: 2段階マトリクスビルド

**根拠**:

1. 拡張バリアント数が5つあり、並列化による時間短縮効果が大きい
2. `needs:` による依存関係は GitHub Actions のネイティブ機能で信頼性が高い
3. 新規拡張追加時の変更箇所がmatrixの1行のみで済む
4. `fail-fast: false` により部分的な成功が可能（安定している拡張はデプロイ継続）

## 影響

### 影響を受けるコンポーネント
- `.github/workflows/docker-publish.yml`: 全面書き換え

### 必要な変更
- 既存の単一ジョブ構成を `build-base` + `build-extensions` の2ジョブに分割
- `build-base` からsha-tagを `outputs` で公開
- `build-extensions` でmatrix strategyを定義
- 各バリアントのタグ生成ロジックを追加

### リスク
- **ベースイメージのビルド失敗**: 全拡張が自動スキップ（`needs:` による）。リスクは低い
- **sha-tag参照の整合性**: 同一ワークフロー実行内のshaを使用するため整合性は保証される
- **GitHub Actions Cache競合**: バリアントごとに `scope` を分離して回避
