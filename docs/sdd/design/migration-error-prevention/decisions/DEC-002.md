# 技術的決定事項 DEC-002: 起動時チェックの実装方針

## ステータス

**承認済** (2026-02-17)

## コンテキスト

スキーマ同期（`drizzle-kit push`）を自動化しても、以下のケースでスキーマ不一致が発生する可能性があります：

1. **手動でのスキーマ変更**: データベースファイルを直接編集
2. **同期失敗**: `drizzle-kit push`が途中で失敗
3. **古いデータベースファイル**: バックアップから復元した場合

これらのケースを検出するため、サーバー起動時にスキーマ整合性チェックを実行する必要があります。

## 検討した選択肢

### オプション1: server.ts統合（推奨）

**概要**: `server.ts`のHTTPサーバー起動前にスキーマ検証を実行

**実装位置**:
```typescript
// server.ts
async function startServer() {
  console.log('🔍 スキーマ整合性チェック中...');

  const validationResult = validateSchemaIntegrity(db.$client);
  if (!validationResult.valid) {
    console.error(formatValidationError(validationResult));
    process.exit(1);
  }

  // HTTPサーバー起動
  const server = app.listen(port, () => {
    console.log(`🚀 サーバー起動: http://localhost:${port}`);
  });
}
```

**メリット**:
- ✅ すべての起動パスで実行される（CLI、Systemd、Docker等）
- ✅ HTTPサーバー起動前にエラー検出（不完全な状態でリクエストを受けない）
- ✅ 実装がシンプル（1箇所に集約）
- ✅ `/api/health`との統合が容易（同じ検証ロジックを再利用）

**デメリット**:
- ❌ サーバー起動シーケンスがわずかに遅延（+70ms程度）

---

### オプション2: CLI起動時のみ実行

**概要**: `src/bin/cli.ts`の`setupDatabase()`内で検証

**実装位置**:
```typescript
// src/bin/cli.ts
export async function setupDatabase() {
  ensureDatabaseFile(databaseUrl);
  syncSchema(databaseUrl);

  // 検証を追加
  const validationResult = validateSchemaIntegrity(db.$client);
  if (!validationResult.valid) {
    console.error(formatValidationError(validationResult));
    process.exit(1);
  }
}
```

**メリット**:
- ✅ CLI起動時のみ実行（サーバー起動時のオーバーヘッドなし）
- ✅ スキーマ同期直後に検証（整合性を即座に確認）

**デメリット**:
- ❌ Docker Compose環境では`node dist/server.js`で直接起動するため、CLIの検証（syncSchema）は実行されない
- ❌ CLIを経由しない起動では検証が漏れるリスク

---

### オプション3: ミドルウェアとして実装

**概要**: Next.jsミドルウェアで初回リクエスト時に検証

**実装位置**:
```typescript
// src/middleware.ts
export function middleware(request: NextRequest) {
  if (!schemaValidated) {
    const result = validateSchemaIntegrity(db.$client);
    if (!result.valid) {
      return NextResponse.json({ error: 'Schema error' }, { status: 503 });
    }
    schemaValidated = true;
  }
  return NextResponse.next();
}
```

**メリット**:
- ✅ サーバー起動自体は成功（HTTPリスナーは起動）
- ✅ 初回リクエスト時のみ検証（遅延初期化）

**デメリット**:
- ❌ 起動時にエラーを検出できない（Fail Fast原則に反する）
- ❌ 初回リクエストのレイテンシが増加
- ❌ 外部監視ツールが正常と誤認する可能性
- ❌ 実装が複雑（グローバル変数管理、並行リクエスト対応）

---

### オプション4: 起動前スクリプトとして分離

**概要**: 独立したスクリプト`check-schema.ts`を作成し、起動前に実行

**実装位置**:
```bash
# package.json
"scripts": {
  "start": "node dist/check-schema.js && node dist/server.js"
}
```

**メリット**:
- ✅ 関心の分離（検証ロジックが独立）
- ✅ CI/CDでの単独実行が容易

**デメリット**:
- ❌ 2つのプロセス起動が必要（オーバーヘッド）
- ❌ Systemd設定が複雑化（ExecStartPre追加が必要）
- ❌ エラー時のログが分散（どちらのプロセスで失敗したか不明瞭）

---

## 決定

**オプション1: server.ts統合 を採用**

### 理由

1. **Fail Fast原則の徹底**
   - HTTPサーバー起動前にエラー検出
   - 不完全な状態でリクエストを受けない
   - Systemdが即座にエラーを検知できる

2. **すべての起動パスで確実に実行**
   - Docker Compose起動（`docker compose up -d`）
   - 手動起動（`node dist/server.js`）
   → どの起動方法でも検証が実行される

3. **外部監視ツールとの統合**
   - SystemdのExecStartPostで`/api/health`を呼び出し
   - サーバー起動前にエラー検出すれば、ヘルスチェック前に停止
   - 監視ツールが正確な状態を把握できる

4. **実装のシンプルさ**
   - 1箇所に集約（`server.ts`のみ変更）
   - 検証ロジックは`validateSchemaIntegrity()`を再利用
   - `/api/health`とコードを共有

5. **性能への影響が軽微**
   - 検証処理は推定70ms程度（7テーブル×PRAGMA table_info、ローカルSQLite読み取り）
   - 実測値は未計測。`PRAGMA table_info`はインメモリSQLite操作のため極めて高速であることを前提とした推定値
   - サーバー起動時の総時間（数秒）に対して無視できる
   - NFR-003（5秒以内）を余裕で達成

### トレードオフ

**受け入れるデメリット**:
- サーバー起動時間がわずかに増加（+70ms）
  → 許容範囲内（NFR-003: 5秒以内）

**軽減策**:
- 検証処理の最適化（必要に応じてキャッシュ導入）
- 並列化（将来的に複数チェックを追加する場合）

### 実装詳細

```typescript
// server.ts
import { validateSchemaIntegrity, formatValidationError } from '@/lib/schema-check';
import { db } from '@/lib/db';

async function startServer() {
  console.log('📦 データベース準備中...');

  // スキーマ整合性チェック
  console.log('🔍 スキーマ整合性チェック中...');
  const validationResult = validateSchemaIntegrity(db.$client);

  if (!validationResult.valid) {
    // 不一致時はエラーメッセージを表示して終了
    console.error(formatValidationError(validationResult));
    process.exit(1);
  }

  console.log('✅ スキーマ整合性OK');
  console.log(`   検証テーブル数: ${validationResult.checkedTables.length}`);

  // HTTPサーバー起動
  const server = app.listen(port, () => {
    console.log(`🚀 サーバー起動: http://localhost:${port}`);
  });

  // ... 以下省略
}
```

## 起動シーケンス

```text
1. データベースファイル初期化
   └─→ ensureDatabaseFile()
        ↓
2. スキーマ同期（CLI起動時のみ）
   └─→ syncSchema() [src/bin/cli.ts]
        ↓
3. スキーマ整合性チェック（全起動パス）★ここで実装★
   └─→ validateSchemaIntegrity() [server.ts]
        ├─→ OK: 次のステップへ
        └─→ NG: エラー表示 + process.exit(1)
        ↓
4. HTTPサーバー起動
   └─→ app.listen()
        ↓
5. WebSocketサーバー起動
```

## 代替案の却下理由

| オプション | 却下理由 |
|-----------|---------|
| CLI起動時のみ実行 | Systemd起動時に検証されないリスク |
| ミドルウェア実装 | Fail Fast原則に反する、初回リクエストの遅延 |
| 起動前スクリプト分離 | 2プロセス起動のオーバーヘッド、Systemd設定複雑化 |
