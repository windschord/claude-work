# DEC-003: network-filter-proxy によるProxy方式ネットワークフィルタリング

## ステータス

**ステータス**: 承認済
**決定日**: 2026-03-12

## コンテキスト

iptablesベースのネットワークフィルタリング（DEC-001）は以下の問題により廃止された（PR#217）:

### 現状の課題
- iptablesルールがホストの他のDockerコンテナにも影響を及ぼす
- HOSTのiptables設定が書き換わってしまう
- Docker Compose環境で`pid: host`, `NET_ADMIN`, `SYS_ADMIN`等の特権設定が必要

### 制約条件
- 既存のUI/API/DB層（ルール管理、テンプレート等）は保持・再利用する
- Docker環境（DOCKER ExecutionEnvironment）のみが対象
- HOST/SSH環境はスコープ外
- network-filter-proxyは外部リポジトリとして管理される

## 情報の明確性

### 明示された情報
- [x] network-filter-proxy (ghcr.io/windschord/network-filter-proxy) のプレビルドイメージを使用
- [x] upstream変更でAPI_BIND_ADDR環境変数を追加し、Docker network経由でAPI通信
- [x] Docker internalネットワークでClaudeコンテナの直接egress不可にする
- [x] 既存のDNS解決機能（resolveHostnames/resolveDomains）は削除する

### 確認状況
- [x] ユーザー確認済み（2026-03-12）

## 検討した選択肢

### 選択肢A: network-filter-proxy（フォワードプロキシ方式）

**概要**: Go製フォワードプロキシをDocker Composeサービスとして起動し、Claudeコンテナからのアウトバウンド通信をプロキシ経由でフィルタリングする。Docker internalネットワークで直接egressを防止。

**メリット**:
- ホストのiptablesに一切影響しない（プロキシプロセス内で完結）
- 他のDockerコンテナに影響しない（独立したネットワーク構成）
- ドメインベースのフィルタリングが可能（DNS解決不要）
- HTTPS CONNECT透過トンネルでTLS通信をMITMせず制御可能
- ルールはインメモリ管理、REST APIによる動的更新が可能（ClaudeWork側で永続化）

**デメリット**:
- upstreamリポジトリへのAPI_BIND_ADDR対応が必要
- proxyサービスが追加されるためリソース消費が増加
- Docker internalネットワーク構成が必要

### 選択肢B: iptables改善（名前空間分離）

**概要**: iptablesルールをコンテナのネットワーク名前空間内に適用し、ホストへの影響を回避する。

**メリット**:
- 追加サービス不要

**デメリット**:
- NET_ADMIN権限が依然必要
- コンテナ名前空間へのアクセスが複雑
- DEC-001の根本問題（特権設定）が解決しない

### 選択肢C: Docker --internal + DNS制御

**概要**: internalネットワーク + カスタムDNSサーバーで名前解決を制御し、許可されていないドメインの解決を拒否する。

**メリット**:
- プロキシ不要で軽量

**デメリット**:
- IPアドレス直接指定による回避が可能
- DNS over HTTPS等のバイパス手段がある
- カスタムDNSサーバーの開発・保守が必要

---

## 決定

**選択**: 選択肢A（network-filter-proxy）

**根拠**:

1. ホストのネットワーク設定に一切影響せず、他コンテナとの干渉がない
2. ドメインベースのフィルタリングにより、DNS解決の複雑性を排除
3. HTTPS CONNECTトンネルによりTLS通信のセキュリティを維持
4. REST APIによる動的ルール管理がClaudeWorkの既存インフラと親和性が高い
5. Docker internalネットワークによるバイパス防止が確実
6. コンテナ起動時のfail-closed: コンテナはinternalネットワークに接続されるため、proxyルール同期前でも直接egressは不可能（ルール未同期状態でも安全）

## 影響

### 影響を受けるコンポーネント
- [NetworkFilterService](../components/network-filter-service.md) @../components/network-filter-service.md: DNS解決関連メソッド削除。ProxyClientはDockerAdapterおよびproxy-syncヘルパーが使用し、NetworkFilterServiceはtestConnectionのhealthCheckでのみProxyClientを使用
- [NetworkFilterUI](../components/network-filter-ui.md) @../components/network-filter-ui.md: フィルタリングステータス表示の更新（proxyヘルスチェック結果表示）
- DockerAdapter: コンテナ作成時にinternalネットワーク接続、HTTP_PROXY環境変数設定
- docker-compose.yml: proxyサービス追加、internalネットワーク定義

### 新規コンポーネント
- [ProxyClient](../components/proxy-client.md) @../components/proxy-client.md: network-filter-proxy Management APIとの通信クライアント

### 必要な変更
- docker-compose.ymlにnetwork-filter-proxyサービス追加
- DockerAdapterのbuildContainerOptionsにネットワーク・プロキシ設定追加
- NetworkFilterServiceからDNS解決メソッド削除、testConnectionにproxy稼働状態確認を追加
- network-filter-proxyへのAPI_BIND_ADDR対応Issue起票

### リスク
- **proxyサービス障害時の影響**: フィルタリングが有効な環境でproxyがダウンすると、該当環境のClaudeコンテナから外部通信不可。フィルタリング無効の環境には影響なし。ヘルスチェックとリカバリ機構で軽減
- **upstream変更の遅延**: API_BIND_ADDRの実装が遅れた場合、暫定的にdocker exec経由でAPI通信するフォールバックを検討
