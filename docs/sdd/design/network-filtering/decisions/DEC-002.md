# DEC-002: ドメイン解決方式 - 起動時DNS解決 + 定期リフレッシュ

## ステータス

承認済

## コンテキスト

iptablesはIPアドレスレベルで動作するが、ユーザーはドメイン名（`*.github.com`等）でルールを定義したい。ドメイン名をIPアドレスに変換する方式を決定する必要がある。

## 検討した選択肢

### A: 起動時DNS解決 + 定期リフレッシュ（採用）

コンテナ起動時にドメイン名をDNS解決してIPアドレスを取得し、iptablesルールに適用する。長時間稼働コンテナにはバックグラウンドで定期的にDNS解決を再実行し、IPアドレスの変更に追従する。

**メリット**:
- 実装がシンプル
- 追加のインフラ（DNSプロキシ等）が不要
- ほとんどのユースケースで十分な精度

**デメリット**:
- CDNなど動的にIPが変わるサービスでは一部のIPが漏れる可能性
- ワイルドカードドメインの全サブドメインを解決することは不可能

### B: カスタムDNSサーバー方式

コンテナのDNSサーバーをカスタムDNSプロキシに設定し、DNS応答を監視してiptablesルールを動的に追加する。

**メリット**:
- 全てのDNS解決結果を捕捉できる
- ワイルドカードドメインの完全な対応が可能

**デメリット**:
- DNSプロキシサーバーの開発・運用が必要
- 複雑性が大幅に増加
- 追加のリソース消費

### C: nftables string match方式

nftablesのpayload inspectionでTLS SNI（Server Name Indication）をマッチングする。

**メリット**:
- カーネルレベルでドメイン名ベースのフィルタリングが可能

**デメリット**:
- nftablesが必要（iptablesとの互換性問題）
- TLS 1.3のECH（Encrypted Client Hello）では使えない
- 設定の複雑性が非常に高い

## 決定

**A: 起動時DNS解決 + 定期リフレッシュ** を採用する。

## 根拠

1. 実装のシンプルさと保守性を最優先
2. Claude Codeの主要な通信先（Anthropic API, npm, GitHub, PyPI）はIPアドレスが比較的安定している
3. CDNのIP変更によるフォールスネガティブは、ユーザーがCIDRブロックを追加することで対応可能
4. 将来的にカスタムDNS方式への段階的移行が可能（API互換を維持）

## 実装詳細

### DNS解決の流れ

1. **通常ドメイン**（例: `api.anthropic.com`）
   - `dns.resolve4()` と `dns.resolve6()` で解決
   - 全ての返却IPアドレスをiptablesルールに追加

2. **ワイルドカードドメイン**（例: `*.github.com`）
   - ベースドメイン（`github.com`）を解決
   - 一般的なサブドメイン（`www`, `api`, `raw`, `gist`等）も解決を試行
   - 解決できたIPアドレスを全てiptablesルールに追加
   - ユーザーには「ワイルドカードは完全なカバレッジを保証しない」旨を通知

3. **IPアドレス/CIDR**（例: `192.168.1.0/24`）
   - DNS解決不要、そのままiptablesルールに適用

### キャッシュ戦略

- メモリ内キャッシュ（Map構造）
- TTL: 5分（デフォルト）
- コンテナ起動時はキャッシュミスの場合のみDNS解決実行
- バックグラウンドリフレッシュ: 10分間隔

### 定期リフレッシュ

- `setInterval` でバックグラウンド実行
- DNS解決結果が前回と異なる場合のみiptablesルールを更新
- リフレッシュ失敗時は既存ルールを維持（警告ログ出力）

## 影響

- コンテナ起動時にDNS解決の分だけオーバーヘッドが発生（通常1秒未満）
- CDN利用サービスでは一部IPが漏れる可能性がある → ドキュメントで制限事項として明記
