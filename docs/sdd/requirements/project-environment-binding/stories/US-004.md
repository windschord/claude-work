# US-004: セッション作成の環境継承

## ユーザーストーリー

**私は** ClaudeWorkのユーザーとして
**〜したい** セッション作成時に環境を気にせず、プロジェクトの環境が自動的に使われるようにしたい
**なぜなら** セッションごとに環境を選択する煩雑さを排除し、1プロジェクト=1環境の明確な関係を維持したいため

## 受入基準（EARS記法）

### AC-001: セッション作成UIの簡素化

- セッション作成モーダルにおいて、システムは環境選択ドロップダウンを表示してはならない
- セッション作成モーダルでは、プロジェクトの環境名を情報として表示してもよい（任意）

### AC-002: セッション作成APIの環境決定

- セッション作成APIが呼び出された時、システムはリクエストボディのenvironment_idパラメータを無視しなければならない
- システムはプロジェクトのenvironment_idを使用してセッションを作成しなければならない
- セッションのenvironment_idカラムにはプロジェクトのenvironment_idと同じ値を保存しなければならない

### AC-003: WebSocket接続時の環境解決

- WebSocket接続時、システムはセッションのプロジェクトのenvironment_idを使用して環境を解決しなければならない
- 現行の4段階フォールバックロジックを、プロジェクトのenvironment_id直接参照に簡素化しなければならない

### AC-004: レガシーパラメータの廃止

- セッション作成APIのdocker_modeパラメータを無視しなければならない（後方互換性のため受け付けるがログで非推奨警告）
