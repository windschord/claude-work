# US-002: コンテナ起動時のフィルタリング自動適用

## ユーザーストーリー

**私は** ClaudeWork管理者として
**〜したい** Dockerコンテナ起動時にネットワークフィルタリングルールが自動的に適用されてほしい
**なぜなら** 手動でルールを適用する手間を省き、設定漏れによるセキュリティリスクを排除するため

## 受入基準（EARS記法）

### 機能要件

- **REQ-002-001**: DockerAdapterがコンテナを起動する時、システムは該当環境のフィルタリング設定を読み込み、有効であればネットワーク制限を適用しなければならない
- **REQ-002-002**: フィルタリングが有効な環境でコンテナを起動する時、システムはDockerカスタムネットワークを作成し、コンテナをそのネットワークに接続しなければならない
- **REQ-002-003**: コンテナ起動時、システムはiptablesルールでデフォルトの外部通信をDROP（拒否）し、ホワイトリストに登録されたドメイン/IPへの通信のみACCEPT（許可）しなければならない
- **REQ-002-004**: ホワイトリストにドメイン名が含まれる場合、システムはDNS解決を行い、解決されたIPアドレスに対してiptablesルールを適用しなければならない
- **REQ-002-005**: ワイルドカードドメイン（例: `*.npmjs.org`）が含まれる場合、システムはDNS解決時にサブドメインを含むIPアドレスを許可しなければならない
- **REQ-002-006**: コンテナ停止・削除時、システムは該当コンテナに関連するiptablesルールを自動的にクリーンアップしなければならない
- **REQ-002-007**: もしiptablesルールの適用に失敗した場合、システムはエラーをログに記録し、コンテナ起動を中止しなければならない
- **REQ-002-008**: システムはDNS解決のためにDNS通信（UDP/TCP 53番ポート）を常に許可しなければならない
- **REQ-002-009**: もしフィルタリングが無効な環境の場合、システムは従来通りネットワーク制限なしでコンテナを起動しなければならない

## データ要件

- iptablesルールの管理にはchainのネーミング規則が必要（例: `CLAUDEWORK-<environment-id>`）
- コンテナIDとiptablesルールの対応付けを管理

## 備考

- iptablesルール適用にはホスト側でroot権限またはCAP_NET_ADMIN権限が必要
- DNS解決結果はTTLに基づきキャッシュし、定期的に更新する仕組みが望ましい（設計フェーズで詳細検討）
- コンテナ起動のオーバーヘッドを最小限に抑える設計が必要
