# 保守性要件

> 非機能要件カテゴリ: 保守性

## 概要

ネットワークフィルタリング機能の保守性に関する要件。コードの品質、テスタビリティ、運用性を確保する。

## 要件一覧

### NFR-MNT-001: テストカバレッジ

**EARS記法**: ネットワークフィルタリング機能のサービス層は、ユニットテストカバレッジ80%以上を維持しなければならない

**測定基準**:
- 指標: サービス層のテストカバレッジ
- 目標値: 80%以上
- 測定方法: vitest --coverage

**関連する機能要件**: 全REQ

**優先度**: 中

**検証方法**:
- [ ] iptablesルール生成ロジックのユニットテストが存在すること
- [ ] ルールCRUD操作のユニットテストが存在すること
- [ ] Docker Compose環境検出のユニットテストが存在すること

---

### NFR-MNT-002: 既存コードへの影響最小化

**EARS記法**: システムはネットワークフィルタリング機能を既存のDockerAdapter・EnvironmentServiceの拡張として実装し、既存のインターフェースを破壊してはならない

**測定基準**:
- 指標: 既存テストの破壊数
- 目標値: 0
- 測定方法: 既存テストスイートの全パス確認

**関連する機能要件**: REQ-002-001, REQ-002-009

**優先度**: 高

**検証方法**:
- [ ] 既存のDockerAdapter関連テストが全てパスすること
- [ ] 既存のEnvironmentService関連テストが全てパスすること
- [ ] フィルタリング無効時の動作が変更前と同一であること

---

### NFR-MNT-003: ログとトラブルシューティング

**EARS記法**: システムはネットワークフィルタリングの適用・解除・エラーをWinstonロガー経由で記録しなければならない

**測定基準**:
- 指標: フィルタリング関連操作のログ出力率
- 目標値: 100%（全操作にログ出力）
- 測定方法: フィルタリング操作実行後のログ確認

**関連する機能要件**: REQ-002-007

**優先度**: 中

**検証方法**:
- [ ] ルール適用時にINFOレベルのログが出力されること
- [ ] ルール適用失敗時にERRORレベルのログが出力されること
- [ ] コンテナ停止時のクリーンアップがログに記録されること
