# セキュリティ要件

> 非機能要件カテゴリ: セキュリティ

## 概要

ネットワークフィルタリング機能のセキュリティに関する要件。フィルタリングの信頼性、バイパス防止、障害時の安全性を確保する。

## 要件一覧

### NFR-SEC-001: デフォルト拒否ポリシー

**EARS記法**: フィルタリングが有効な環境において、システムはホワイトリストに明示的に許可されていない全ての外部通信をデフォルトで拒否しなければならない

**測定基準**:
- 指標: ホワイトリスト外の通信ブロック率
- 目標値: 100%
- 測定方法: テスト用コンテナからホワイトリスト外のアドレスへの通信テスト

**関連する機能要件**: REQ-002-003

**優先度**: 高

**検証方法**:
- [ ] フィルタリング有効環境でホワイトリスト外アドレスへのcurl/wgetがブロックされること
- [ ] DNS解決自体は許可されるが、実際の通信がブロックされること

---

### NFR-SEC-002: フィルタリングバイパス防止

**EARS記法**: システムはコンテナ内からのiptablesルール変更やネットワーク設定変更によるフィルタリングのバイパスを防止しなければならない

**測定基準**:
- 指標: コンテナ内からのフィルタリング回避成功率
- 目標値: 0%
- 測定方法: コンテナ内からiptablesコマンド実行、ネットワーク設定変更を試行

**関連する機能要件**: REQ-002-003, REQ-002-002

**優先度**: 高

**検証方法**:
- [ ] コンテナにCAP_NET_ADMIN権限が付与されていないこと
- [ ] コンテナ内からiptablesコマンドが実行不可であること
- [ ] コンテナ内からのネットワークインターフェース設定変更が不可であること

---

### NFR-SEC-003: 障害時のフェイルセーフ

**EARS記法**: もしフィルタリングルールの適用に失敗した場合、システムはコンテナを制限なしで起動するのではなく、起動を中止しなければならない

**測定基準**:
- 指標: フィルタリング失敗時の制限なし起動回数
- 目標値: 0回
- 測定方法: iptablesコマンドを意図的に失敗させた状態でコンテナ起動を試行

**関連する機能要件**: REQ-002-007

**優先度**: 高

**検証方法**:
- [ ] iptablesコマンドが利用不可な環境でフィルタリング有効のコンテナ起動がエラーとなること
- [ ] エラーメッセージが適切に表示されること

---

### NFR-SEC-004: ルール変更時のクリーンアップ

**EARS記法**: コンテナ停止時、システムは該当コンテナに関連する全てのiptablesルールを確実にクリーンアップしなければならない

**測定基準**:
- 指標: 停止済みコンテナに関連する残存iptablesルール数
- 目標値: 0
- 測定方法: コンテナ停止後にiptables -Lでルール残存を確認

**関連する機能要件**: REQ-002-006

**優先度**: 高

**検証方法**:
- [ ] コンテナ正常停止後にiptablesルールが残存しないこと
- [ ] コンテナ異常終了後もクリーンアップが実行されること
- [ ] アプリケーション再起動時に古いルールがクリーンアップされること
