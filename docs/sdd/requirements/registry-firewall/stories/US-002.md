# US-002: コンテナからのレジストリプロキシ利用

## ユーザーストーリー

**私は** ClaudeWorkのDocker実行環境のユーザーとして
**~したい** パッケージインストール時に自動的にregistry-firewallを経由したい
**なぜなら** 悪意あるパッケージのインストールを未然に防止するため

## 受入基準(EARS記法)

### 機能要件

- **REQ-004**: registry-firewallが有効な時、Docker実行環境のコンテナ起動時にシステムは各パッケージマネージャーのレジストリURLをregistry-firewallに設定しなければならない
  - npm: `npm config set registry http://claudework-registry-firewall:8080/npm/`
  - pip: `PIP_INDEX_URL=http://claudework-registry-firewall:8080/pypi/simple/`
  - go: `GOPROXY=http://claudework-registry-firewall:8080/go/,direct`
  - cargo: Cargo設定ファイルでレジストリURLを設定
  - docker: Docker daemon設定でmirror URLを設定(該当する場合)

- **REQ-005**: registry-firewallが有効な時、各パッケージマネージャーのレジストリ設定はコンテナの環境変数またはパッケージマネージャーの設定ファイルで注入しなければならない
  - 環境変数で設定可能なもの(pip, go)は環境変数を使用
  - 設定ファイルが必要なもの(npm, cargo)はentrypoint scriptまたはDockerイメージ内で設定

- **REQ-006**: システムは既存のnetwork-filter-proxyとregistry-firewallを共存させなければならない
  - パッケージレジストリへのアクセス: registry-firewall経由
  - その他のHTTP/HTTPSアクセス: network-filter-proxy経由(既存動作を維持)
  - network-filter-proxyのホワイトリストにregistry-firewallの内部通信を追加不要(同一ネットワーク内)

## 画面/UI要件

なし(バックエンド設定のみ)

## データ要件

### コンテナ環境変数(registry-firewall有効時に追加)

| 変数名 | 値 | 対象 |
|--------|-----|------|
| `PIP_INDEX_URL` | `http://claudework-registry-firewall:8080/pypi/simple/` | pip |
| `PIP_TRUSTED_HOST` | `claudework-registry-firewall` | pip(HTTP接続許可) |
| `GOPROXY` | `http://claudework-registry-firewall:8080/go/,direct` | go |

### コンテナ起動時設定スクリプト

npmとCargoは環境変数だけでは設定できないため、コンテナ起動時にスクリプトで設定:
- npm: `npm config set registry http://claudework-registry-firewall:8080/npm/`
- cargo: `~/.cargo/config.toml`にレジストリ設定を書き込み

## 備考

- registry-firewallが無効の場合、これらの環境変数/設定は注入しない(直接レジストリにアクセス)
- registry-firewallのURLはDockerのサービス名解決を利用(`claudework-registry-firewall`)
- HTTP接続を使用(同一Docker network内のため暗号化不要)
- npmレジストリのサポートはregistry-firewall Issue#23で対応予定。config.yamlにnpmの設定を含めているが、registry-firewall側の対応が完了するまでnpmプロキシ経由のリクエストは直接アップストリームにフォールバックされる可能性がある
